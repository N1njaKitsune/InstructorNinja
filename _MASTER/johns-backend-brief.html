Proposed Schema Approach

For Review

Our recommendations on schema design. If you're happy with these, build against them. If not, let's talk it through.

A1 Element Stripes: Separate belt_progression Table

▶
Recommendation: Separate table, not JSON on student.

One row per student per belt level. Columns: student_id, belt_id, air, earth, water, fire, element_5 (nullable — see A7).

Keeps history naturally across belt levels. Cleaner queries for grading and profile display.

A2 Aura Stage: Integer on Student Record

▶
Recommendation: Single integer field on the student table.

aura_stage INTEGER DEFAULT 1 — range 1 to 13. Updated only by Nendo session completions. Grading does NOT affect it.

Simple linear progression, doesn't warrant its own table. Can be migrated to a table in Stage 2 if Nendo gets complex.

A5 Grading: Two Tables (Events + Awards)

▶
Recommendation: Separate grading_events and grading_awards tables.

grading_events = one row per grading session (class, instructor, date).

grading_awards = one row per stripe awarded (student, element, value, FK to event).

Both tables are immutable (Rule #9). Corrections use a supersedes FK (see C1 below).

A7 Fifth Element: Nullable Column Now

▶
Recommendation: Plan for 5 elements. Add nullable element_5 column to belt_progression.

Column stays NULL until the 5th element is designed and named (expected 6-12 months). UI should be flexible for 4-5 badges. Near-zero cost now, avoids retrofitting later.

B1 RBAC: Application-Level Enforcement

▶
Recommendation: Field-level access control at the application layer, not middleware.

Each API endpoint/serializer explicitly includes or excludes fields based on the user's role. Three roles: Student, Parent (Guardian), Instructor. Access matrix defined in the Shared Data Model spec (Section 6).

Faster to build, easier to debug. The access matrix is simple enough that app-level enforcement is sufficient.

B4 Avatar Placeholder: Ninja Silhouette

▶
Recommendation: Generic ninja silhouette SVG as placeholder. Full avatar system designed before launch.

A simple ninja character icon renders when no photo is set. SVG asset — no external dependency. Full avatar customisation is a later feature (logged in Improvement Log).

Proposed API & Data Flow Approach

For Review

C1 Grading Corrections: Supersedes FK

▶
Recommendation: Corrections are new rows with a supersedes_award_id FK. Originals are never modified.

When a mistake is found, create a new grading_awards row with correction_type = 'revoke' and supersedes_award_id pointing to the original. The system reads the latest non-superseded award as truth. Full audit trail preserved (Rule #9).

C2 Grading Receipts: Server-Generated

▶
Recommendation: The POST /api/grading-events endpoint returns the receipt in its response.

One source of truth — the receipt always matches what the database recorded. The instructor app displays what comes back, no client-side receipt generation.

C3 Belt Promotion: Automatic on Challenge Confirm

▶
Recommendation: When parent confirms challenge, server promotes belt in the same transaction.

One API call from the parent app: POST /api/challenges/{id}/confirm. Server sets challenge_confirmed = true, creates belt award event, updates student status to active with new belt, sends notifications — all in one transaction. No separate promotion endpoint needed.

Still Open — Needs Resolution

3 Items

These are unresolved. You can start around them but they'll need answering before the relevant tables are finalised.

A3 Soft Delete Convention

▶
OPEN: Should all tables use a status enum, a deleted_at timestamp, or both?

Affects every table. Student already uses a status enum. Options: extend that pattern to all tables, use a universal deleted_at, or use both (deleted_at for soft delete, status for business meaning).

A4 Student-to-Family Relationship

▶
OPEN: One family per student (simple FK) or many-to-many (junction table for blended families)?

Affects parent schema, notification routing, and billing. One-to-many covers 95%+ of cases. Many-to-many properly handles shared custody but adds complexity everywhere.

A6 Challenge Schema

▶
OPEN: Standalone challenges table or fields directly on the student record?

Standalone table preserves challenge history across belts. Fields on student are simpler but lose history when belt changes. Related to Working Doc Pending Decision #10.
