# InstructorNinja v4.8 — Comprehensive Review

## Overview

v4.8 is a substantial single-file HTML prototype (~1,900 lines) covering Home, Classes, Directory, Student Manager, Overview, and Manage (with 9 sub-views). It works well as a demo, but organic growth across 8 versions has introduced significant duplication and structural debt. This review catalogues every area worth addressing before cutting a v5.0.

---

## 1. Code Duplication

### 1.1 Finance rendering — almost identical functions

`renderBilling()` (line ~1443) and `renderOvFinance()` (line ~1470) produce the same output — same hardcoded values, same HTML structure, same alerts. The only differences are the target element and minor label tweaks ("271 students × £120" vs "271 × £120").

**Impact:** ~120 lines of near-identical code.
**Fix:** Extract a single `buildFinanceHTML()` function, call it from both locations.

### 1.2 Belt distribution rendering — duplicated in 3 places

The `bd()` utility renders belt bars. But `buildSD()` and `buildOv()` each independently compute `tot` (the school-wide belt totals) and `mx` (the max), then call `bd()`. The computation is identical.

**Impact:** Minor, but three copies of the same aggregation logic.
**Fix:** Single `getSchoolBeltTotals()` function returning `{totals, max}`.

### 1.3 Growth chart — identical in school detail and overview

`buildSD()` and `buildOv()` both call `gb()` for the growth chart and render it into their respective containers with the same surrounding markup.

**Fix:** Shared builder function.

### 1.4 Summary card grids — repeated pattern

The `.g3` / `.g4` + `.scard` pattern (value + label + optional subtitle) appears in: Home, Classes overview, School detail, Enrolments, Waitlist, Trials, Follow-up, Staff, Compliance, Schedule, and Overview Stats. Each is built by hand with raw HTML strings.

**Impact:** Dozens of repetitions of the same card markup.
**Fix:** A `statCard(value, label, subtitle?)` helper that returns the HTML.

### 1.5 Detail view pattern — list → detail → success

Waitlist, Trials, Follow-up, Staff, Compliance, and Enrolments all follow the exact same pattern:
1. Summary cards at top
2. List of items (`.card` with click handler)
3. Detail view (back button + `.irow` fields)
4. Action buttons
5. Success animation (`.success-wrap` with setTimeout → return to list)

Each is implemented from scratch with its own `render___()` and `show___Detail()` functions. The success state alone (`success-wrap` div → setTimeout → goManage) is copy-pasted 5 times.

**Impact:** This is the single largest source of duplication. Roughly 400+ lines could be reduced to ~150 with a shared detail-view renderer.
**Fix:** Generic `renderListView({title, summaryCards, items, onItemClick})` and `renderDetailView({backFn, title, sections, actions})` builders.

### 1.6 Back-bar / breadcrumb header — 4 variants

There are four near-identical header bar patterns:
- `.brow` (light background) — used in class detail, school detail, belt list, promo, summary, receipt, student manager, and every manage detail view
- `.dbrow` (dark background) — used in register, grade roster, grade screen
- Inline `<div class="brow">` within `innerHTML` strings in manage views
- `mtop` in modals

They all do the same thing: back button + title + optional subtitle. But each is styled slightly differently and built independently.

**Fix:** Single `buildHeader(title, subtitle, backFn, dark?)` function.

### 1.7 Receipt rendering — duplicated in class detail and overview

`renderCD()` and `renderOvHist()` both contain nearly identical receipt card markup (`.receipt` with `.rec-hdr`, `.rec-els`, element icons). The inline element summary (`EL[ei].n: X full, Y half`) is duplicated verbatim.

**Fix:** Extract `buildReceiptCard(receipt, onClick)`.

## 2. CSS Issues

### 2.1 Redundant / overlapping classes

- `.card` and `.scard` are identical except `.scard` exists to distinguish stat cards semantically — but they share all properties.
- `.trow` (tab row) and `.ov-tabs` are the same pattern with different class names.
- `.tbtn` and `.ov-tab` are functionally identical.
- `.sec-lbl` and `.glbl` (directory group label) share most properties.
- `.hist-item` / `.hist-col` / `.hist-dot` / `.hist-line` are duplicates of `.tl-item` / `.tl-col` / `.tldot` / `.tl-line` — the timeline component was built twice (once for belt promo, once for student history).

**Impact:** ~40 unnecessary CSS rules.
**Fix:** Merge into shared utility classes.

### 2.2 Inline styles overwhelming the CSS system

The prototype has a solid CSS foundation, but many renders inject heavy inline styles instead of using classes. For example, `renderCD()` builds performance bars, stat rings, and enrolled student rows entirely with inline `style=` attributes rather than reusable classes.

Rough estimate: 30-40% of visual styling is inline rather than in the stylesheet.

**Fix:** Move repeated inline patterns to named classes.

### 2.3 Unused or dead CSS

- `.dot-F` (future attendance dot) is defined but only used in the seed data; it's the same as `.bdr`.
- `.grow.sel` and `.grow:hover:not(.sel)` both resolve to `background:var(--dk)` — the hover state does nothing.
- `.btn-full` is used in some places but many full-width buttons just use `style="width:100%"` instead.

### 2.4 CSS variable under-use

The colour palette in `:root` is well-defined but many render functions use raw hex values instead of variables. For example `#639922` (green) appears 15+ times in JS but is never assigned a CSS variable. Same for `#BA7517`, `#378ADD`, `#FF314A`.

**Fix:** Add `--green`, `--amber`, `--blue-acc`, `--red-acc` variables and use them consistently.

## 3. JavaScript Architecture

### 3.1 Hardcoded values scattered throughout

- `271` (total students) appears as a literal in: home, classes overview banner, school detail, overview stats, billing, term management, and the enrolment summary. Should be computed from `STUDS.length` or `CLS.reduce()`.
- `120` (term fee) is hardcoded in renderBilling, renderOvFinance, and renderTerm. Should come from `TERMS[0].fee`.
- `'Harry Bovill'` is hardcoded in 8+ places. Should come from a `CURRENT_USER` constant.
- `'Spring 2026'`, `'Week 5 of 12'` appear as string literals in multiple tab headers.

### 3.2 Cryptic function names

Single-letter or abbreviated names make the code hard to follow:
- `cc()` → capacity colour
- `bd()` → belt distribution
- `bm()` → belt mini
- `gb()` → growth bars
- `usc()` → update selection count
- `ubadges()` → update badges
- `pip()` → belt pip/chip
- `sr()` → seeded random
- `tgPend()` → toggle pending
- `upui()` → update UI

These are fine for a compressed prototype but will hinder maintenance. A v5 should use readable names.

### 3.3 The `gen()` function — 1 massive line

The student data generator (`gen()`, line ~989) is a single line of ~600 characters. It's unreadable and unmaintainable. Same for `genCompliance()`.

**Fix:** Break into readable multi-line functions.

### 3.4 Global state management

All state is in loose global variables: `curTab`, `curCls`, `cv`, `regSt`, `grSel`, `staged`, `receipts`, `pending`, `rmTgt`, `sumFrom`, `curManage`, `dirSort`, `curStudent`, `studentReturnCtx`, `homeDay`. There's no central state object or update pattern.

**Fix:** Group into a single `STATE` object. Consider a simple pub/sub or at minimum a `setState()` wrapper.

### 3.5 innerHTML string building

Almost every render function builds HTML via string concatenation or template literals assigned to `el.innerHTML`. This:
- Makes the code fragile (one missing quote breaks everything)
- Prevents attaching event listeners cleanly (relies on `onclick="..."` attributes)
- Creates XSS risk if any user-entered data (e.g. notes) is not escaped

**Fix:** While a full framework is overkill for a prototype, a simple `h(tag, attrs, children)` helper would reduce errors significantly. At minimum, escape user input in notes.

### 3.6 No input sanitisation

`addNote()` inserts raw text into innerHTML. If a note contains HTML characters, it will be interpreted as markup.

**Fix:** Escape `<`, `>`, `&`, `"` in all user inputs before rendering.

## 4. Structural / UX Refinements

### 4.1 Manage tab has too many sub-views in one tab row

The Manage tab has 9 buttons in a single `.trow`: Enrolments, Waitlist, Trials, Follow-up, Schedule, Staff, Compliance, Cover, Term. On a 390px screen, this overflows and likely requires horizontal scrolling (which isn't styled).

**Fix:** Group into categories (e.g. "Students" → Enrolments/Waitlist/Trials/Follow-up; "Operations" → Schedule/Staff/Cover; "Admin" → Compliance/Term). Or use a grid/list selector instead of a scrolling tab bar.

### 4.2 Role-based visibility is incomplete

`applyRole()` hides the Manage nav tab for non-admins and hides Overview for assistants. But: the Finance tab in Overview still renders billing data visible to all roles. Staff pay information is gated on `curRole==='admin'` in the profile view but the staff list itself is visible to all. The Compliance view shows all student documents regardless of role.

**Fix:** Audit every view against the ROLES permission model.

### 4.3 Grading flow state isn't persisted

Register marks (`regSt`) are stored in a plain object and lost on page refresh. Same for staged stripes, grading selections, and notes. For a prototype this is expected, but worth noting for v5 — even localStorage persistence would improve demo usability.

### 4.4 Attendance data is synthetic and static

The `att` array is generated once on load and never updates when the register is used. Marking a student present/absent in the register doesn't update their attendance history or percentage.

**Fix:** Wire register confirmation to update `s.att` and `s.ap`.

### 4.5 Dead code / legacy references

- `renderBilling()` targets `#admp-billing` which no longer exists in the HTML (the old admin tab structure was removed). The function is never called.
- Comment on line 659: "INST_MAP and TERM_DATES declared below in v4.4 Administration section" — accurate but indicates code was patched in rather than properly integrated.
- Line 1407: "Legacy admin tab code removed" comment but `renderBilling()` still exists.

---

### PROMPT — InstructorNinja v5.0 Refactor

You are refactoring `NinjaLearning_InstructorApp_v4.8.html` into `v5.0`. This is a single-file HTML prototype for an instructor-facing martial arts school management app. It runs in a 390×844 mobile shell.

**Goal:** Produce a v5.0 that is functionally identical to v4.8 but significantly cleaner, more maintainable, and ~30% smaller. No features should be removed. No visual changes unless noted.

**Read MANIFEST.md before starting.** Follow the prototype lifecycle: archive v4.8 reference, update MANIFEST, create the new file.

#### Phase 1 — Shared utilities (do first)

Create these JS helper functions at the top of the `<script>` block:

1. **`statCard(value, label, subtitle?)`** — Returns the HTML for a `.scard` stat card. Replace all manual stat card markup throughout.

2. **`buildHeader(title, subtitle, backFn, opts?)`** — Returns a `.brow` or `.dbrow` header bar. `opts.dark` toggles dark mode. `opts.rightContent` allows a right-side button/badge. Replace all hand-built back-bar markup.

3. **`buildDetailView({backFn, title, subtitle?, sections[], actions[]})`** — Renders a standard detail view. Each `section` is `{label, content}` where content is either an array of `{label, value}` rows (rendered as `.irow`) or raw HTML. Each `action` is `{label, style, onClick}`. Replace the detail views in: waitlist, trials, follow-up, staff profile, student compliance, enrolment detail.

4. **`showSuccess(message, subtitle, returnFn, delay=1500)`** — Renders the `.success-wrap` pattern and auto-returns. Replace the 5+ copy-pasted success states.

5. **`buildFinanceHTML(targetStudents, termFee, collected, pending, overdue)`** — Single finance renderer. Replace both `renderBilling()` (delete it — the target element doesn't exist) and `renderOvFinance()`.

6. **`buildReceiptCard(receipt, onClickFn)`** — Returns receipt card HTML. Replace duplicate receipt rendering in `renderCD()` and `renderOvHist()`.

7. **`escapeHTML(str)`** — Escapes `<>&"` for safe innerHTML insertion. Apply to all user-generated content (notes, etc.).

#### Phase 2 — CSS cleanup

1. **Merge duplicate classes:**
   - Delete `.ov-tabs` and `.ov-tab` — use `.trow` and `.tbtn` instead.
   - Delete `.hist-item`, `.hist-col`, `.hist-dot`, `.hist-line` — use `.tl-item`, `.tl-col`, `.tldot`, `.tl-line` instead (update `renderSpHistory` to use the `tl-` classes).
   - Merge `.card` and `.scard` if truly identical, or keep `.scard` as a modifier.

2. **Add CSS variables** for colours used heavily in JS:
   ```css
   --green-acc: #639922;
   --amber-acc: #BA7517;
   --blue-acc: #378ADD;
   --red-acc: #FF314A;
   ```
   Replace hardcoded hex values in JS render functions with `var(--green-acc)` etc. (in inline styles this means using the variable name in the style string).

3. **Move repeated inline style patterns** to named classes. Target the biggest offenders: performance bars in `renderCD()`, stat rings, directory rows.

4. **Remove dead CSS:** `.grow.sel` / `.grow:hover` no-ops, unused `.dot-F` if redundant.

#### Phase 3 — Constants and state

1. **Replace hardcoded values:**
   - `271` → compute from `CLS.reduce((a,c) => a + c.enrolled, 0)` or `STUDS.length` (they should agree)
   - `120` (term fee) → `TERMS.find(t => t.status === 'current').fee`
   - `'Harry Bovill'` → `CURRENT_USER.name` (define a constant)
   - `'Spring 2026'` / `'Week 5 of 12'` → derive from `TERMS`

2. **Group globals** into a `STATE` object:
   ```js
   const STATE = { tab: 'home', classId: 4, classView: 'day', regMarks: {}, gradeSelection: new Set(), staged: [], pending: [], ... };
   ```

3. **Rename cryptic functions** (keep old names as one-line aliases if needed for onclick handlers):
   - `cc()` → `capacityColor()`
   - `bd()` → `beltDistribution()`
   - `bm()` → `beltMini()`
   - `gb()` → `growthBars()`
   - `pip()` → `beltChip()`
   - `sr()` → `seededRandom()`
   - `usc()` → `updateSelectionCount()`
   - `ubadges()` → `updateBadgeCounts()`

#### Phase 4 — Readability

1. **Break `gen()`** into a multi-line, readable function with named intermediate variables.

2. **Break `renderCD()`** — the longest render function — into sub-functions: `renderClassSummary()`, `renderClassPerformance()`, `renderClassStudents()`, `renderClassHistory()`.

3. **Add section comments** between major blocks (home screen, classes screen, directory, student manager, overview, manage).

#### Phase 5 — Manage tab UX

1. **Add `overflow-x: auto; -webkit-overflow-scrolling: touch;`** to the Manage tab's `.trow` so the 9 buttons scroll cleanly on mobile.

2. Optionally: group the 9 sub-views into 3 categories with section headers inside a scrollable list, rather than a single overflowing tab bar. Only do this if it can be done without losing functionality.

#### Constraints

- **Single HTML file.** No external dependencies, no build step.
- **Functionally identical.** Every screen, flow, and interaction in v4.8 must work the same in v5.0.
- **Same visual design.** No colour, typography, or layout changes unless explicitly noted above.
- **Same data.** Keep all mock data (students, classes, belts, receipts, etc.) producing the same output.
- **Delete `renderBilling()`** — its target element (`#admp-billing`) doesn't exist. The finance view lives in Overview now.
- Do **not** run `git push` or `git commit` — this is a sub-project workspace.
