# UI & Prototypes

> **Department type:** Active
> **Owner:** Hazza (Design Lead)
> **Division:** Design & Frontend

---

## Purpose

This department is responsible for all visual and interactive outputs: screens, wireframes, and interactive HTML prototypes. It translates design decisions and specs into tangible, reviewable artefacts that the build team can implement from.

Everything a user will see or touch passes through this department first.

---

## Specialist Skills

These are available to this department in addition to the shared tools (`docx`, `file-reading`, `pdf`):

| Skill | Purpose |
|-------|---------|
| `frontend-design` | HTML/CSS/JS prototyping, responsive layouts, component design |
| `pptx` | Presentation decks for walkthroughs, design reviews, stakeholder demos |

---

## Scope

**In scope:**
- Interactive HTML prototypes (Student, Parent, Instructor, Admin views)
- Wireframes and screen flows
- UI specification documents
- Presentation decks for design reviews
- Visual QA of prototypes against specs

**Out of scope:**
- UX philosophy and age-design rationale (→ Creative Design + Ergonomics)
- Backend schema or API work (→ Backend & Deployment)
- Server or deployment configuration (→ Infrastructure & Security)

## Sub-Agents (Planned)

As the department matures, the following sub-agents may be introduced:

- **Prototype Builder** — generates and iterates on HTML prototypes from UI specs
- **Visual QA Agent** — compares prototypes against specs and flags discrepancies

These do not exist yet. They will be created via the `skill-creator` skill when the workflow demands it.

---

## Reporting

This department reports to the Design Lead (Hazza). All design decisions must be recorded in the Working Doc before the session ends, per the dual-Claude sync protocol in `_WORKING_DOCS/CLAUDE.md`.
